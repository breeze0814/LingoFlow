# 轻量桌面翻译器技术选型

配套执行文档见：[文档总览](./文档总览.md)

## 1. 推荐结论
这次技术选型的核心判断只有一句：
`V1 先做 macOS，架构保留跨平台扩展点。`
推荐技术路线：
- `桌面宿主`：Tauri 2
- `核心语言`：Rust
- `前端`：React 19 + TypeScript
- `构建工具`：Vite
- `样式`：CSS Modules + CSS Variables
- `本地 HTTP`：Axum + Tokio
- `上游服务请求`：Reqwest + Serde
- `配置存储`：Tauri Store
- `密钥存储`：系统 Keychain
- `日志`：Tracing
- `macOS 原生桥接`：Swift Helper
- `V1 OCR`：Apple Vision
这套组合优先满足 4 个目标：
- 轻量
- 稳定
- 快速落地
- 未来可扩展
## 2. 平台策略
V1 不建议全平台首发。
原因不是框架做不到，而是我们的核心功能都强依赖平台能力：
- 划词翻译依赖前台应用选中文本读取
- 截图 OCR 依赖截图权限和屏幕能力
- 截图翻译依赖截图链路和 OCR 链路打通
如果一开始追求 macOS、Windows、Linux 同时完整交付，复杂度会立刻集中在：
- 全局快捷键
- 选中文本获取
- 截图权限
- OCR 实现
- 浮层窗口行为
这会让首版偏离“轻量工具”目标，变成“跨平台兼容工程”。
因此平台策略定为：
- `产品交付`：V1 先做 macOS
- `架构设计`：统一接口，后续再补 Windows / Linux Adapter
## 3. 核心技术栈
### 3.1 Tauri 2
选择 `Tauri 2` 作为桌面宿主。
理由：
- 相比 Electron 更轻，更符合常驻工具定位
- Rust 原生能力接入自然
- 官方有全局快捷键、Store 等成熟插件
- 单窗口工具型应用是它的适配场景
不选 Electron：
- 运行时更重
- 资源占用更高
- 对当前产品没有足够收益
### 3.2 Rust
选择 `Rust` 作为核心编排层。
Rust 负责：
- 任务编排
- Provider 调用
- 本地 HTTP 服务
- 配置读写
- 错误模型
- 日志
- 调用平台桥接层
理由：
- 与 Tauri 组合成熟
- 性能和内存占用适合桌面常驻工具
- 类型系统适合收敛 Provider、状态和错误边界
### 3.3 React 19 + TypeScript
选择 `React 19 + TypeScript` 作为 UI 层。
理由：
- 资料和生态最成熟
- 设置页、结果面板、状态处理足够稳
- 维护成本低
- 对单主面板工具来说运行时负担可接受
不选 Vue / Svelte / Solid：
- 不是因为它们不行
- 而是当前阶段更看重资料密度和稳定维护
### 3.4 Vite
选择 `Vite` 作为前端构建工具。
理由：
- 开发启动快
- 与 Tauri 集成顺手
- 适合小型桌面工具产物组织
### 3.5 CSS Modules
V1 不引入大型 UI 组件库，不使用 Tailwind 作为主方案。
采用：
- `CSS Modules`
- `CSS Variables`
理由：
- 依赖少
- 控制力高
- 结果面板和设置页规模很小，不需要完整设计系统
明确不选：
- Ant Design
- MUI
- 大型桌面组件平台
## 4. 本地能力方案
### 4.1 macOS 平台桥接
推荐增加一个 `Swift Helper`，只负责 macOS 平台相关能力：
- 读取前台应用选中文本
- 调用截图能力
- 调用 Apple Vision OCR
Rust 只负责调度 Helper 和统一结果。
这样划分的好处是：
- 平台问题集中在平台层
- 业务逻辑仍然留在 Rust
- UI 不需要接触权限和原生 API 细节
### 4.2 划词翻译
V1 通过 `Accessibility API` 获取选中文本。
推荐链路：
- 全局快捷键触发
- Rust 调用 Swift Helper
- Helper 读取选中文本
- Rust 进入翻译流程
- UI 展示结果
### 4.3 截图能力
截图能力分层：
- `Tauri` 负责窗口和结果面板
- `Swift Helper` 负责真实截图和屏幕相关能力
这样做能避免把权限、屏幕坐标和浮层行为混在前端里。
### 4.4 OCR 能力
V1 的 OCR 默认选 `Apple Vision`。
理由：
- macOS 自带能力
- 无需用户单独申请 OCR 服务
- 延迟更低
- 很适合截图 OCR 和截图翻译首版链路
虽然底层是 Apple Vision，但业务层仍实现成统一的 `OcrProvider` 接口，后续可替换为云 OCR 或其他平台 OCR。
## 5. 服务与存储
### 5.1 本地 HTTP
本地 HTTP 服务采用 `Axum + Tokio`。
理由：
- 路由清晰
- 错误处理模型清楚
- 本地 localhost API 测试方便
- 与 Rust 异步生态天然匹配
V1 只保留最小 API：
- `POST /translate`
- `GET /selection_translate`
- `GET /input_translate`
- `GET /ocr_recognize`
- `GET /ocr_translate`
### 5.2 上游 Provider 请求
翻译和云 OCR 统一采用：
- `Reqwest`
- `Serde`
候选翻译服务清单见：[翻译 API 清单](./翻译API清单.md)。
理由：
- Rust 生态主流组合
- JSON、超时、连接复用能力成熟
- 适合做统一 Provider 抽象
### 5.3 配置存储
V1 的非敏感配置使用 `Tauri Store`。
存储内容：
- 快捷键
- 默认目标语言
- 启用的服务
- 窗口位置和大小
- 普通 UI 偏好
不选 SQLite 作为 V1 主存储：
- 当前数据量太小
- 先上数据库属于过度设计
### 5.4 密钥存储
API Key 不进入普通配置文件，统一放系统 Keychain。
原则：
- 配置和密钥分离
- 普通配置可读可迁移
- 密钥单独托管
## 6. 模块边界
推荐模块：
- `frontend`
- `core`
- `platform/macos`
- `providers`
- `http-api`
- `storage`
职责边界：
- 前端不直接接平台能力
- 平台层不直接调翻译服务
- Provider 不管理 UI 状态
- HTTP API 复用同一套 orchestrator
## 7. 测试策略
推荐测试组合：
- `Rust 单元测试`：orchestrator、provider、错误模型
- `Rust 集成测试`：localhost API
- `Vitest + React Testing Library`：结果面板和设置页
V1 不做：
- 完整桌面端 E2E 自动化
- 视觉回归体系
首版先保证主链路可靠，再逐步补自动化广度。
## 8. 明确不选
- `Electron`
- `Flutter`
- `Qt`
- `SQLite 作为 V1 主存储`
- `大型 UI 组件库`
- `全平台同时首发`
## 9. 风险判断
最大风险不是翻译 API，而是 macOS 原生能力接入：
- 选中文本读取
- 截图权限
- OCR 权限与行为一致性
应对方式：
- 平台问题收敛到 Swift Helper
- 业务问题收敛到 Rust
- 展示问题收敛到 React UI
## 10. 最终建议
最终推荐路线：
- `交付范围`：V1 先做 macOS
- `桌面框架`：Tauri 2
- `核心后端`：Rust
- `前端`：React 19 + TypeScript + Vite
- `样式`：CSS Modules
- `本地 HTTP`：Axum + Tokio
- `服务调用`：Reqwest + Serde
- `配置存储`：Tauri Store
- `密钥管理`：系统 Keychain
- `原生桥接`：Swift Helper + Accessibility API + Apple Vision
这条路线最适合当前这版产品目标：把划词翻译、输入翻译、截图 OCR、截图翻译和本地 HTTP 调用先做顺，而不是一开始把系统做胖。
