# 轻量桌面翻译器配置模型
## 1. 目标
统一定义哪些配置进 Store，哪些密钥进 Keychain，以及配置版本如何迁移。

## 2. 配置分层
- `Store`：普通配置、UI 偏好、窗口状态、端口等非敏感信息。
- `Keychain`：API Key、Secret、Token。

## 5. Keychain 命名约定
- `lingoflow.provider.openai_compatible_ocr.api_key`
- `lingoflow.provider.openai_compatible.api_key`
- `lingoflow.provider.deepl_free.api_key`
- `lingoflow.provider.azure_translator.api_key`
- `lingoflow.provider.google_translate.api_key`
- `lingoflow.provider.tencent_tmt.secret_id`
- `lingoflow.provider.tencent_tmt.secret_key`
- `lingoflow.provider.baidu_fanyi.app_id`
- `lingoflow.provider.baidu_fanyi.secret`

网页源 Provider 当前不要求单独密钥：
- `lingoflow.provider.youdao_web.*`
- `lingoflow.provider.bing_web.*`

命名原则：
- 统一前缀
- 按 Provider 分隔
- 不把明文密钥写进 Store

## 6. 默认值原则
- 默认值必须可直接启动应用。
- 默认值必须保证至少一个翻译 Provider 和一个 OCR Provider 可选。
- 缺少云服务密钥时，应用可以启动，但相关 Provider 标记为未就绪。
- 浏览器预览环境允许退回 `localStorage` 读取非敏感配置。

## 7. 迁移规则
- `version` 为配置结构版本号。
- 任何破坏性修改都必须升级 `version`。
- 迁移逻辑只允许前进，不允许静默回退。
- 无法迁移时必须显式报错并保留原文件。

## 8. 不允许的实现
- 把 API Key 存进普通 JSON。
- 在多个地方维护不同的默认值。
- 让 Provider 自己读取配置文件。
- 用空字符串代表“有效配置”。
