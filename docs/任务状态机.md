# 轻量桌面翻译器任务状态机
## 1. 目标
统一 `selection_translate`、`input_translate`、`ocr_recognize`、`ocr_translate` 的状态流转，避免 UI 和编排层各自维护一套逻辑。

## 2. 任务类型
- `selection_translate`
- `input_translate`
- `ocr_recognize`
- `ocr_translate`
- `open_input_panel`

`open_input_panel` 是 UI 命令，不进入完整翻译链路。

## 3. 全局规则
- V1 只允许一个活跃任务。
- 活跃任务指状态为 `collecting_input` 或 `pending`。
- 新任务启动前必须取消旧任务。
- 已经进入 `success`、`failure`、`cancelled` 的任务不可恢复。
- 任务状态必须由 orchestrator 单点更新。

## 4. 状态定义
- `idle`：当前无任务。
- `collecting_input`：等待用户输入或截图框选。
- `pending`：任务已开始执行，等待外部结果。
- `success`：任务成功结束。
- `failure`：任务失败结束。
- `cancelled`：任务被用户或系统取消。

## 5. 状态流转
### 5.1 直接任务
适用于 `selection_translate`：
`idle -> pending -> success | failure | cancelled`

### 5.2 需要用户补充输入的任务
适用于 `input_translate`、`ocr_recognize`、`ocr_translate`：
`idle -> collecting_input -> pending -> success | failure | cancelled`

### 5.3 仅打开输入面板
适用于 `open_input_panel`：
`idle -> collecting_input -> cancelled`

## 6. 事件定义
- `triggered`：用户或 HTTP 触发任务。
- `input_collected`：输入框已提交或截图已完成。
- `provider_started`：进入 Provider 调用阶段。
- `provider_succeeded`：拿到成功结果。
- `provider_failed`：Provider 返回失败。
- `user_cancelled`：用户取消输入或截图。
- `system_cancelled`：新任务覆盖旧任务，或系统中断。

## 7. UI 行为要求
### 7.1 idle
- 主面板可隐藏或显示上一次成功结果。
- 不显示加载态。

### 7.2 collecting_input
- 显示输入中或截图中的明确提示。
- 不允许重复触发同类任务创建第二个会话。

### 7.3 pending
- 显示统一加载态。
- 显示任务来源和任务类型。
- 禁止再次提交同一任务。

### 7.4 success
- 展示最终结果。
- 展示来源 Provider。
- 允许复制结果和重新触发。

### 7.5 failure
- 展示明确错误信息。
- 保留触发来源，方便用户重试。

### 7.6 cancelled
- 不展示假成功。
- 用户主动取消时不弹错误提示。
- 因系统中断取消时可记调试日志。

## 8. 对 HTTP API 的要求
- 直接任务只有在进入终态后才返回最终结果。
- `open_input_panel` 类命令返回 `accepted`，不等待终态。
- 用户取消截图时返回 `cancelled`，而不是 `500`。

## 9. 对日志的要求
- 每个任务都要有 `task_id`。
- 每次状态变化都要写日志。
- 终态日志必须包含 `task_type`、`duration_ms`、`result`。

## 10. 不允许的实现
- UI 组件自行维护独立状态机。
- Provider 直接改 UI 状态。
- 在失败后偷偷切换到另一个 Provider。
- 把取消当成功处理。
