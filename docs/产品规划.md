# 轻量桌面翻译器产品规划

配套执行文档见：[文档总览](./文档总览.md)

## 1. 产品定位

一句话定义：

做一个轻量、快速、可被外部调用的桌面翻译器，保留 Bob 的核心体验，吸收 Pot 的开放调用能力，但不做臃肿的生态型产品。

目标关键词：

- 触发快
- 结果快
- 配置少
- 单路径完成任务
- 可扩展但不过度设计

## 2. 设计原则

### 2.1 产品原则

- 只解决高频场景，不追求功能堆叠。
- 所有入口最终汇聚到同一个结果工作区模型。
- 默认突出展示一个首选结果，但保留 Provider 结果栈用于切换和对比。
- 设置项尽量少，优先提供合理默认值。
- 能力接口提前抽象，但首版不开放插件系统。

### 2.2 技术原则

- 入口层与能力层解耦。
- 翻译与 OCR 分开建模，不混成单一功能。
- Provider 统一接口，便于后续替换和扩展。
- 所有失败明确暴露，不做静默降级。

## 3. 目标用户

- 需要频繁划词和短文本翻译的桌面用户
- 需要截图识别和截图翻译的用户
- 希望通过外部工具快速触发翻译能力的效率用户

## 4. 核心使用场景

### 4.1 划词翻译

用户在任意应用内选中文本，按快捷键后立即看到翻译结果。

### 4.2 输入翻译

用户通过快捷键呼出输入窗口，输入文本并回车得到翻译结果。

### 4.3 截图 OCR

用户框选屏幕区域，识别出原文，便于复制或后续处理。

### 4.4 截图翻译

用户框选屏幕区域，直接得到翻译结果，不需要先复制文字。

### 4.5 外部调用

用户通过其他工具或脚本触发翻译和 OCR，作为桌面工作流的一部分使用。

## 5. 功能范围

### 5.1 V1 必做功能

#### 入口能力

- 划词翻译
- 输入翻译
- 截图 OCR
- 截图翻译
- 全局快捷键

#### 核心能力

- 自动检测源语言
- 目标语言选择
- 翻译 Provider 接入能力
- OCR Provider 接入能力
- 请求中状态展示
- 错误提示与失败暴露

#### UI 能力

- 设置主窗口
- 独立结果工作窗口
- 原文区与结果区
- 手动重新翻译
- 复制结果
- Provider 结果栈与置顶结果
- 基础加载态与空态

#### 设置能力

- 快捷键设置
- 默认目标语言设置
- 默认翻译 Provider 设置
- 默认 OCR Provider 设置
- 翻译服务配置
- OCR 服务配置
- API Key 与服务参数配置
- 本地 HTTP 端口配置
- 权限状态查看

#### 集成能力

- 本地 HTTP 调用入口
- 最小 API 集合：
- `POST /translate`
- `GET /selection_translate`
- `GET /input_translate`
- `GET /ocr_recognize`
- `GET /ocr_translate`

#### 数据能力

- 本地配置持久化
- 敏感配置写入 Keychain

- 自动复制译文

- 调用超时提示

### 5.3 V2 候选功能

- 多翻译源并行
- 多 OCR 源并行

- 生词本导出

## 6. 明确不做

以下能力不进入首版，避免产品膨胀：

- 插件市场
- 插件在线安装与更新
- 大量内置服务全覆盖
- 多结果瀑布流展示
- 复杂收藏与标签系统
- 账户体系
- 云同步
- 多面板多窗口设计
- 复杂主题系统

## 7. 版本边界

### 7.1 V1 边界

V1 只要求完成最核心的单次任务闭环：

- 能触发
- 能拿到结果
- 能看见状态
- 能复制结果
- 能完成基本配置

V1 不追求生态完整，也不追求服务数量优势。

### 7.2 V1.5 边界

V1.5 解决的是“更顺手”，不是“更多能力”。

重点放在：

- 降低重复配置成本
- 提升连续使用效率
- 提升错误可理解性

### 7.3 V2 边界

V2 才考虑“扩展性”和“开放性”升级，但一次只上一个大能力。

优先级建议：

1. 多服务并行
2. 插件系统

不要在同一个版本同时引入这两项，以免系统复杂度陡增。

## 8. 首版服务策略

### 8.1 翻译服务

当前实现已接入：

- `youdao_web`
- `bing_web`
- `deepl_free`
- `azure_translator`
- `google_translate`
- `tencent_tmt`
- `baidu_fanyi`

### 8.2 OCR 服务

当前实现已接入：

- 本地 OCR
: macOS 下使用 `apple_vision`，Windows 下使用 `tesseract_js_ocr`
- `openai_compatible_ocr`

## 9. 信息架构建议

### 9.1 模块划分

- `entry`
- `orchestrator`
- `providers`
- `ui`
- `storage`

### 9.2 模块职责

- `entry` 负责划词、输入、截图、HTTP 调用的触发。
- `orchestrator` 负责接收任务、调度 Provider、更新状态。
- `providers` 负责翻译和 OCR 服务适配。
- `ui` 负责结果展示和设置界面。
- `storage` 负责配置与轻量历史存储。

## 10. 验收标准

### 10.1 V1 验收标准

- 任意文本可通过划词翻译完成一次完整翻译。
- 用户可通过输入窗口完成一次完整翻译。
- 用户可通过截图完成一次 OCR。
- 用户可通过截图完成一次翻译。
- 失败时界面明确提示错误。
- 用户可完成服务配置并保存。
- 外部脚本可通过本地 HTTP 成功触发翻译或 OCR。
- `GET /input_translate` 可成功拉起输入工作区。

## 11. 当前结论

当前产品路线确定为：

- 先做轻量主流程
- 先把 4 个高频入口做顺
- 先保留本地 HTTP 调用能力
- 主窗口承载设置，结果在独立工作区完成
- macOS 为正式支持平台，Windows 为实验支持
- 明确推迟插件与生态型功能

这份文档作为后续需求拆分和技术设计的基础版本。
